# Taller de Métricas para la Gestión de Proyectos de Software

Este repositorio contiene un ejercicio autoguiado para medir el avance, calidad, eficiencia y costos de un mini-proyecto de software usando GitHub Projects, GitHub Issues, GitHub Actions y un dashboard web.

## Objetivo

Aplicar métricas de gestión de proyectos de software para tomar decisiones basadas en datos.

## Métricas analizadas

- Porcentaje de tareas completadas.
- Velocity del sprint.
- Bugs abiertos y cerrados.
- Tiempo promedio de resolución.
- Horas estimadas vs. horas reales.
- Variación de presupuesto.
- Tareas vencidas.
- Riesgos del proyecto.

## Análisis de resultados

### 1. Avance del proyecto
El proyecto va al 60% de avance (3 de 5 tareas cerradas), evidenciando un estado **medio**. Quedan 2 tareas pendientes por completar.

### 2. Velocity del sprint
Se completaron 6 story points de 9 planeados. La velocity cubre la mayor parte del alcance pero no la totalidad, por lo que es **insuficiente** para cerrar el sprint completo.

### 3. Bugs abiertos
Hay 1 bug abierto y ninguno cerrado, lo que representa un riesgo activo para la entrega. La calidad del producto se considera **en riesgo**.

### 4. Horas reales vs estimadas
Las horas reales igualaron exactamente las estimadas (10h), sin ninguna desviación de esfuerzo.

### 5. Desviación de costo
No hubo desviación de costo. El estimado y el real fueron exactamente **$200 USD**, con un ROI del 200%.

### 6. Tareas de mayor riesgo
El bug abierto es la tarea de mayor riesgo activo. El nivel de riesgo general del proyecto es **Medio**.

### 7. Decisión del jefe de proyecto
Se recomienda priorizar el cierre del bug antes de continuar con las tareas restantes, y revisar las 2 tareas abiertas para garantizar la entrega del sprint.

### 8. Métrica adicional
Se agregaría el **porcentaje de tareas cerradas a tiempo** para mejorar la visibilidad del cumplimiento del sprint frente a las fechas planeadas.
